# Quickstart: 验证三项修复

本文件给出手动验证三个修复的最短路径，供实现完成后自测与回归。前置：已装 git，
已在 GitView 添加至少一个可用账号并同步过远程仓库。

---

## 构建与静态检查

```bash
# 前端
pnpm install
pnpm lint            # eslint --max-warnings 0
pnpm exec prettier --check "src/**/*.{ts,vue}"

# 后端（src-tauri）
cargo fmt --check
cargo clippy -- -D warnings
cargo test           # 分支映射 / 策略映射等单元测试

# 联调
pnpm tauri dev
```

## 验证问题① — Git 设置生效

1. 进入「设置 → Git」，把「默认 pull 策略」改为 `变基（--rebase）`，保存。
2. 打开一个本地仓库工作区，点 **Pull**。
3. 到「操作日志」查看该次 Pull 记录，**命令字段应为 `git pull --rebase`**（而非
   `--ff-only`）。
4. 把「默认 push 策略」改为 `current`，Push 后日志命令应体现 `push.default=current`。
5. **Git 可执行路径**：在「设置 → Git」手动指定一个有效 git 路径并保存，执行任意
   Git 操作应仍成功（走指定路径）；清空后回退 PATH 自动探测。
6. **提交身份**：在一个 `user.name`/`user.email` 均未配置的仓库中，于「设置 → Git」
   填写身份并保存，提交面板应 **不再提示「请在设置中配置 Git 身份」**，提交成功后
   `git log` 作者为设置中的身份。若仓库/全局已配置身份，则以既有 config 为准。

**通过标准**：设置修改后无需重启，对应 Git 操作即按新策略/路径/身份执行；操作日志
命令字段如实反映所用策略。

## 验证问题② — 克隆时选择分支

1. 「远程仓库」多选若干仓库 → 「批量 Clone」。
2. 对话框中每个仓库行出现 **分支选择器**，默认选中各自默认分支；下拉可搜索、列出
   该仓库的远程分支。
3. 对某仓库选择一个非默认分支，确认克隆。
4. 克隆完成后打开该仓库，`git branch` / `git status` 应处于所选分支。
5. 未改动分支的仓库仍克隆默认分支（行为不变）。
6. **降级**：分支列表拉取失败时，选择器显示占位并允许「使用默认分支」继续克隆，
   不阻断整个批量任务。

**通过标准**：每仓库可独立选分支，克隆结果落在所选分支；缺省与失败均安全回退默认分支。

## 验证问题③ — 变更列表完整展示与折叠

1. 构造一个有大量变更的仓库（如新增 50+ 文件），打开工作区。
2. 左侧「文件变更」的「已暂存」「未暂存」两段：
   - **分组内部可滚动**，能看到该段下的**全部**文件（不再被裁剪、无内容丢失）。
   - 点击分组标题可**折叠/展开**该段。
3. 折叠一段后另一段获得更多可视空间；展开恢复。
4. 窗口缩小时列表区不撑破布局，滚动条正确出现在分组内部。

**通过标准**：任意数量的变更文件都可通过滚动完整查看；分组可折叠；会话内折叠状态保持
（重开工作区可恢复默认展开）。

## 回归关注点

- Pull `ff_only` 默认路径行为与现状一致（未改设置的用户无感知）。
- HTTPS 克隆凭据注入、代理注入、取消/重试、skipped 判定均不受 `--branch` 影响。
- 提交前其余 4 项校验（detached HEAD / 冲突 / 无暂存 / message 空）逻辑不变。
- 操作日志脱敏行为不变。
