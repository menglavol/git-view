# Feature Specification: 修复设置生效、克隆选分支与变更列表展示

**Feature Branch**: `002-fix-settings-clone-diff`

**Created**: 2026-07-06

**Status**: Draft

**Input**: User description: "分析并修复三个问题：1、设置部分的某些配置项不生效（如「默认 Pull 策略」）；2、远程仓库克隆时无法选择分支；3、本地仓库详情页「已暂存/未暂存」没有折叠控件、文件过多时无法展示全部文件。"

## 背景与问题定位摘要 *(供评审参考，非需求本体)*

三个问题分别对应三条独立价值链，可独立开发、测试与验收：

- **问题①（设置不生效）** — 根因：设置能正确保存到本地存储，但**执行 Git 操作时并不读取这些配置**。以「默认 Pull 策略」为例：无论用户在设置里选哪一项，实际 pull 始终以「仅快进」方式执行。同类还包括「默认 Push 策略」「Git 可执行文件路径」「提交身份 user.name / user.email」——它们都属于「配置只被保存、从未被消费」的同一类缺陷。（本次范围：整个 Git 设置组。）
- **问题②（克隆不能选分支）** — 根因：批量克隆配置对话框没有分支选择项，克隆流程也不支持指定分支，因此每个仓库只能克隆其默认分支。
- **问题③（变更列表展示不全 / 无折叠）** — 根因：「已暂存 / 未暂存」两段列表区域没有滚动能力，内容随文件数无限撑高并被父容器裁剪，文件多时（如 76 个）看不到全部；同时分组标题不可点击折叠（用户所称「下拉框」）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Git 设置项真正生效 (Priority: P1)

用户在「设置 → Git」中修改配置（默认 Pull 策略、默认 Push 策略、Git 可执行文件路径、提交身份 user.name / user.email）并保存后，期望这些配置在后续实际的 Git 操作中被真实采用，而不是被程序内部的固定行为覆盖。

**Why this priority**: 这是「所见即所得」的信任基础问题——设置界面提供了选项却不生效，会让用户误以为操作已按其意图执行（例如以为在 rebase，实际却在快进失败），可能导致误判仓库状态。属于正确性缺陷，优先级最高。

**Independent Test**: 可独立验证——分别把 Pull 策略设为「变基」「普通合并」「仅快进」，在一个与远端已分叉的仓库上执行 Pull，观察实际行为是否随设置改变；把 Push 策略、Git 路径、提交身份设为特定值后执行对应操作并核对结果。无需依赖问题②③。

**Acceptance Scenarios**:

1. **Given** 用户将「默认 Pull 策略」设为「变基（--rebase）」并保存，**When** 在一个本地与远端已分叉的仓库执行 Pull，**Then** 系统以变基方式合并远端更新（而非报「无法快进」错误）。
2. **Given** 用户将「默认 Pull 策略」设为「普通合并」，**When** 在分叉仓库执行 Pull，**Then** 系统允许产生 merge commit 完成合并。
3. **Given** 用户将「默认 Pull 策略」设为「仅快进」，**When** 在分叉仓库执行 Pull，**Then** 系统返回「无法快进合并」的中文友好提示（保持原有安全行为）。
4. **Given** 用户在设置里指定了自定义 Git 可执行文件路径并通过校验保存，**When** 执行任意 Git 操作（status / pull / push / commit 等），**Then** 系统使用该自定义路径的 Git，而非仅依赖系统 PATH。
5. **Given** 用户在设置里填写了提交身份 user.name / user.email 并保存，**When** 在一个尚未配置提交身份的仓库执行提交，**Then** 提交能成功完成且提交作者为设置中填写的身份（不再因「未配置身份」被拦截）。
6. **Given** 用户将「默认 Push 策略」设为某一具体策略并保存，**When** 执行 Push，**Then** 推送行为符合所选策略。

### User Story 2 - 变更文件列表可完整查看与折叠 (Priority: P1)

用户打开本地仓库工作区，左侧「已暂存 / 未暂存」两段文件列表在文件数量很多时（几十到数百个）应能完整查看全部文件，并能折叠 / 展开任一分组以聚焦另一分组。

**Why this priority**: 当前有大量变更的仓库（截图中未暂存 76 个）根本无法看到、无法操作被裁剪掉的文件，直接阻断了「暂存 / 丢弃 / 查看 diff」核心工作流，是可用性阻断级缺陷。改动局部、风险低，与 P1 并列。

**Independent Test**: 可独立验证——打开一个有大量未提交变更的仓库，确认可通过滚动看到并操作全部文件；点击分组标题可折叠 / 展开该分组。无需依赖问题①②。

**Acceptance Scenarios**:

1. **Given** 某仓库未暂存变更有 76 个文件，**When** 用户打开该仓库工作区，**Then** 用户可通过滚动查看并操作到列表中的每一个文件（包括最后一个）。
2. **Given** 「已暂存」与「未暂存」两段同时有较多文件，**When** 用户点击「未暂存」分组标题，**Then** 该分组内容折叠收起、标题保留计数，「已暂存」分组不受影响；再次点击恢复展开。
3. **Given** 用户折叠了某分组，**When** 用户切换文件过滤条件或触发状态刷新，**Then** 列表数据正确更新且不会因折叠状态导致文件丢失或无法操作。
4. **Given** 某分组为空，**When** 展示该分组，**Then** 显示「无」占位且分组标题计数为 0。

### User Story 3 - 克隆时选择分支 (Priority: P2)

用户在「远程仓库」多选若干仓库发起批量克隆时，可为每个待克隆仓库选择要克隆的分支（默认选中该仓库的默认分支），克隆结果即为所选分支。

**Why this priority**: 属于能力增强而非阻断缺陷——不选分支时仍可克隆默认分支，现有主流程不受影响；但能显著提升「只需要某个特性分支」场景的效率。故次于两个正确性 / 可用性阻断问题。

**Independent Test**: 可独立验证——多选仓库打开批量克隆对话框，为某仓库选择非默认分支后开始克隆，克隆完成后核对该仓库检出的分支为所选分支。无需依赖问题①②。

**Acceptance Scenarios**:

1. **Given** 用户在批量克隆对话框中查看某仓库，**When** 展开其分支选择控件，**Then** 系统从对应平台拉取并展示该仓库的分支列表，且默认选中该仓库的默认分支。
2. **Given** 用户为某仓库选择了非默认分支，**When** 开始克隆，**Then** 克隆完成后该仓库检出的当前分支为用户所选分支。
3. **Given** 用户未改动任何分支选择，**When** 开始克隆，**Then** 各仓库均按各自默认分支克隆（与现有行为一致）。
4. **Given** 某仓库分支列表拉取失败（网络 / 权限问题），**When** 用户查看该仓库分支控件，**Then** 系统给出可理解的中文提示并允许回退为「克隆默认分支」，不阻断其他仓库的克隆。
5. **Given** 批量选中多个仓库，**When** 用户为不同仓库分别选择不同分支，**Then** 每个克隆任务各自使用其对应仓库所选的分支。

### Edge Cases

- **设置项**：设置里填写的 user.name / user.email 与目标仓库已有的仓库级配置冲突时，应有明确、可预期的优先级规则（见 FR-005）。
- **设置项**：Pull 策略为「变基」时若产生冲突，应复用现有「用外部工具解决冲突」的中文提示，而非静默失败。
- **设置项**：自定义 Git 路径在保存后变为不可用（被删除 / 移动），Git 操作应给出可理解的错误而非崩溃。
- **变更列表**：单个分组文件数达到数百至上千时，滚动查看仍应保持界面可响应（不长时间卡死）。
- **变更列表**：折叠状态是否在同一次工作区会话内保持（切换文件、刷新状态后）——需明确（见 Assumptions）。
- **克隆分支**：仓库分支数量极多（数百分支）时，分支选择控件应可用（支持搜索 / 过滤，避免下拉过长不可用）。
- **克隆分支**：分支列表拉取有网络开销，需避免在对话框打开时对所有选中仓库一次性并发拉取造成明显卡顿（见 FR-013）。
- **克隆分支**：选定分支后若该分支在克隆前于远端被删除，克隆应失败并给出可理解的中文提示，不影响其他任务。

## Requirements *(mandatory)*

### Functional Requirements

**问题① — Git 设置项生效（整个 Git 设置组）**

- **FR-001**: 执行 Pull 时，系统 MUST 依据用户当前保存的「默认 Pull 策略」（仅快进 / 变基 / 普通合并）决定合并方式，而非使用固定行为。
- **FR-002**: 执行 Push 时，系统 MUST 依据用户当前保存的「默认 Push 策略」执行推送。
- **FR-003**: 执行任意 Git 操作时，若用户已在设置中指定并通过校验保存了自定义 Git 可执行文件路径，系统 MUST 使用该路径的 Git；未指定时回退系统 PATH 中的 Git。
- **FR-004**: 当用户在设置中保存了提交身份 user.name / user.email 后，系统 MUST 使这些值在提交时真正生效（提交作者为该身份），并使「提交前身份校验」在已于设置中配置身份的情况下通过。
- **FR-005**: 系统 MUST 对「设置中的提交身份」与「仓库已有 Git 身份配置」定义并遵循明确的优先级：以仓库级已有配置优先（若存在），仅在仓库未配置身份时采用设置中的身份填补——避免覆盖用户为特定仓库刻意设置的身份。
- **FR-006**: 上述所有设置项 MUST 在保存后立即对后续操作生效，无需重启应用。
- **FR-007**: 设置项生效改造 MUST 保留现有的中文友好错误提示（如 Pull 无法快进、冲突、Push 被拒 / 无 upstream / 权限不足）与敏感信息脱敏行为。

**问题② — 克隆时选择分支**

- **FR-010**: 批量克隆配置对话框 MUST 为每个待克隆仓库提供分支选择控件，默认选中该仓库的默认分支。
- **FR-011**: 分支选择控件 MUST 展示对应仓库在其平台（GitHub / GitLab / Gitee）上的分支列表供用户选择。
- **FR-012**: 克隆流程 MUST 支持按每个任务所选分支进行克隆，克隆完成后仓库检出的当前分支为所选分支；用户未改选时按默认分支克隆（保持现有行为）。
- **FR-013**: 分支列表 MUST 按需 / 惰性拉取（如展开某仓库分支控件时才拉取），避免打开对话框即对所有选中仓库一次性拉取造成明显卡顿；已拉取结果宜在本次对话框会话内缓存复用。
- **FR-014**: 当某仓库分支列表拉取失败时，系统 MUST 给出可理解的中文提示，并允许该仓库回退为「克隆默认分支」，且 MUST NOT 阻断其他仓库的分支拉取与克隆。
- **FR-015**: 批量克隆时，各任务 MUST 各自独立记录并使用其所选分支（不同仓库可选不同分支）。

**问题③ — 变更文件列表完整展示与折叠**

- **FR-020**: 「已暂存 / 未暂存」两段文件列表 MUST 在文件数量超出可视区域时提供滚动，使用户可查看并操作到每一个文件。
- **FR-021**: 「已暂存」与「未暂存」分组标题 MUST 可点击折叠 / 展开，且折叠某分组不影响另一分组。
- **FR-022**: 分组标题 MUST 持续显示该分组当前文件计数（含空分组显示 0）。
- **FR-023**: 折叠 / 滚动改造 MUST 不破坏现有功能：状态过滤、单击查看 diff、单击暂存 / 取消暂存、全部暂存 / 全部取消暂存、右键丢弃及其二次确认均照常工作。
- **FR-024**: 变更列表在大量文件（数百量级）下 MUST 保持界面可响应，滚动与操作不出现明显卡顿。

### Key Entities *(include if feature involves data)*

- **Git 设置（Git Settings）**：用户在「设置 → Git」维护的一组配置，含默认 Pull 策略、默认 Push 策略、Git 可执行文件路径、提交身份 user.name / user.email。本特性的核心变化是让这些属性在**操作执行阶段被真实读取与应用**。
- **克隆任务（Clone Task）**：一次仓库克隆的描述。本特性为其新增「目标分支」属性——每个任务记录其要克隆的分支（缺省为对应仓库默认分支）。
- **远程仓库分支（Remote Branch）**：某远程仓库在其平台上的分支集合，用于在克隆对话框中供用户选择；每个仓库有一个默认分支作为初始选中项。
- **文件变更（File Change）**：工作区中某文件的变更项（路径 + 状态 + 是否已暂存），按「已暂存 / 未暂存」分组展示；本特性改变的是其**展示与折叠 / 滚动方式**，数据结构不变。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对「默认 Pull 策略」的三种取值，在同一分叉仓库上执行 Pull 的实际行为与所选策略 100% 一致（3/3 场景通过）。
- **SC-002**: 在未预先配置提交身份的仓库中，仅通过设置界面填写提交身份即可成功提交，无需再手动执行任何命令行配置，用户完成一次提交的操作路径不增加额外步骤。
- **SC-003**: Git 设置组中「有界面选项」的每个配置项均可被验证为「实际生效」，不存在「界面可改但操作不采用」的项（该类缺陷数降为 0）。
- **SC-004**: 在未暂存变更为 76 个文件的仓库中，用户可查看并操作到 100% 的文件（含最后一个），此前为无法查看被裁剪部分。
- **SC-005**: 用户可在 1 次点击内折叠或展开任一变更分组，折叠仅影响目标分组。
- **SC-006**: 在数百个变更文件的仓库中滚动变更列表时，界面保持流畅响应（无可感知卡顿）。
- **SC-007**: 用户可在批量克隆前为选中仓库选择分支，克隆完成后检出分支与所选分支 100% 一致；未改选时全部按默认分支克隆。
- **SC-008**: 单个仓库分支列表拉取失败时，其余仓库的克隆成功率不受影响（失败隔离，不产生连带失败）。

## Assumptions

- **范围边界**：本特性聚焦三个具体问题。问题①的修复范围为**整个 Git 设置组**（Pull/Push 策略、Git 路径、提交身份），不含对「通用 / 网络 / 外部工具 / 账号与安全」其余标签页配置项生效情况的全面审计（如需可另开特性）。
- **提交身份优先级**：采用「仓库级已有配置优先，设置身份仅在仓库未配置时填补」的规则（FR-005），以免覆盖用户为特定仓库刻意设置的身份；此为合理默认，若业务希望「设置身份始终覆盖」需另行确认。
- **克隆选分支粒度**：采用**每仓库可选、分支列表从平台 API 实时拉取、默认选中默认分支**的方案；分支列表惰性拉取以控制开销。
- **分支控件可用性**：当仓库分支数量很大时，分支选择控件应支持搜索 / 过滤以保持可用（沿用项目既有大数据量列表处理理念）。
- **变更列表展示**：采用**可折叠分组 + 分组内部滚动**方案，不引入虚拟滚动；若后续出现上千文件的极端仓库导致卡顿，再评估引入虚拟滚动（复用远程 / 本地仓库列表既有方案）。
- **折叠状态持久化**：折叠 / 展开状态默认仅在当前工作区会话内保持，不跨仓库切换或重启持久化（除非后续另有要求）。
- **兼容与安全**：所有改造复用项目既有的中文错误提示、敏感信息脱敏、危险操作二次确认等既定约束，不降低现有安全与体验标准。
- **依赖**：问题②依赖各平台（GitHub / GitLab / Gitee）提供可用的「仓库分支列表」获取能力；实现细节留待规划阶段。
