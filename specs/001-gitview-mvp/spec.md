## User Scenarios & Testing *(mandatory)*

### User Story 1 - 多平台多账号统一管理 (Priority: P1)

作为开发者，我需要在 GitView 中统一添加并管理 GitHub、GitLab、Gitee 三大平台的多个账号
（包括公司自建 GitLab 实例），通过 Personal Access Token 验证连接，从而在一个客户端
内访问所有平台的仓库资源。

**Why this priority**: 账号管理是 GitView 一切能力的基础前提；没有账号就无法同步、浏览、
克隆任何远程仓库。该故事独立交付即可让用户开始"账号集中存放"的差异化体验，是
其他所有 P1/P2 故事的依赖前置。

**Independent Test**: 通过依次添加一个 GitHub 公开账号、一个自建 GitLab 账号、一个 Gitee 账号，
观察"测试连接"是否返回正确的用户信息（用户名、头像、显示名）；删除账号后凭据应
从安全存储中彻底清除。该测试无需依赖远程仓库同步即可独立完成。

**Acceptance Scenarios**:

1. **Given** 用户首次启动 GitView 且未添加任何账号，**When** 选择"添加 GitHub 账号"
   并填入有效 Personal Access Token，**Then** 系统应在 10 秒内返回连接成功，并
   显示该账号的用户名、显示名、头像。
2. **Given** 用户已有 1 个 GitHub 账号，**When** 添加第 2 个 GitHub 账号（不同用户名），
   **Then** 系统应正确并列两个账号，不互相覆盖；用户可标记其中一个为默认账号。
3. **Given** 用户填入了无效的 Token，**When** 点击"测试连接"，**Then** 系统应给出
   明确的错误提示（如"Token 无效或已过期"），且不保存任何凭据。
4. **Given** 用户配置自建 GitLab 地址 `https://git.company.com:8443`，**When** 点击
   测试连接，**Then** 系统应正确推导 API 地址 `https://git.company.com:8443/api/v4`
   并校验成功，允许用户手动修改 API 地址。
5. **Given** 用户使用内网 GitLab 地址 `http://gitlab.local`，**When** 保存账号，
   **Then** 系统应弹出"使用明文 HTTP 连接"的安全提示，由用户确认后才允许保存。
6. **Given** 用户对自建 GitLab 实例遇到自签名证书错误，**When** 系统提示证书不受信任，
   **Then** 用户可在该实例的高级设置中勾选"允许该实例使用自签名证书"，重新测试
   连接成功。
7. **Given** 用户删除一个已添加账号，**When** 确认删除，**Then** 该账号在列表中
   消失，对应的 Token 凭据从系统安全存储中删除，且本地已克隆仓库记录保留。

### User Story 2 - 远程仓库统一浏览与搜索 (Priority: P1)

作为同时管理多个平台仓库的开发者，我希望在一个统一的列表中浏览自己所有账号下的
远程仓库，并通过搜索、筛选、收藏等能力快速找到目标仓库，从而摆脱在多个网页之间
来回切换的困扰。

**Why this priority**: 远程仓库统一浏览是"多平台统一管理"差异化能力的可视化呈现，
也是 US3（批量 Clone）的前置数据来源。用户只有先看到仓库列表，才能选择克隆。

**Independent Test**: 在已添加至少一个账号的前提下，执行"同步仓库"，验证列表中
出现该账号下所有可见仓库；输入仓库名片段验证搜索过滤；切换"仅显示已收藏"过滤
器验证收藏标记持久化。该测试无需依赖批量 Clone 即可独立验证浏览体验。

**Acceptance Scenarios**:

1. **Given** 用户已添加 1 个 GitHub 账号且该账号下有 50 个仓库，**When** 点击"同步
   仓库"，**Then** 系统应在显示进度（如"已同步 30/50"）的同时分页拉取，最终列表
   完整呈现 50 个仓库。
2. **Given** 同步过程中网络中断或 API 限流，**When** 错误发生，**Then** 系统应保留
   旧缓存数据不被清空，并显示"同步失败，已保留上次缓存"提示。
3. **Given** 远程仓库列表中包含 GitHub、GitLab、Gitee 多平台仓库共 1000 条，**When**
   用户在搜索框输入 "user-service"，**Then** 列表应在 500 ms 内过滤出所有名称或
   描述匹配的仓库。
4. **Given** 用户在筛选器中选择"仅显示私有仓库"且"账号 = 公司 GitLab"，**When** 应用
   筛选，**Then** 列表应只显示该账号下可见性为 private 的仓库。
5. **Given** 用户对若干仓库标记收藏，**When** 切换到"仅显示已收藏"视图，**Then**
   列表仅显示收藏过的仓库；关闭并重新启动应用后收藏状态仍保留。
6. **Given** 用户点击某个仓库的"详情"按钮，**When** 侧边详情抽屉打开，**Then** 抽屉
   中应显示仓库名、描述、所属账号、默认分支、可见性、网页地址、HTTPS / SSH 克隆地址、
   语言、最近更新时间；若该仓库已克隆到本地，还应显示本地路径。

### User Story 3 - 多仓库批量克隆 (Priority: P1)

作为需要批量拉取项目的开发者（如新入职、跨团队协作、批量备份），我希望从远程
仓库列表中多选若干仓库，一次性下发克隆任务，由系统自动按可控并发执行并展示
每个任务的实时进度，从而节省"逐个复制 URL、逐个执行 git clone"的大量重复操作。

**Why this priority**: 批量 Clone 是 GitView 区别于 GitHub Desktop、SourceTree 等
传统客户端的**最核心差异化能力**，也是产品定位中明确强调的"核心价值"之一。该
故事独立交付即可让用户感受到 GitView 的核心价值主张。

**Independent Test**: 在已同步至少 5 个仓库的前提下，多选其中 5 个仓库，配置目标
目录与并发数为 3，执行批量 Clone；观察任务队列展示 5 个任务，并发执行不超过 3 个，
任务最终全部成功后自动加入本地仓库列表，目标目录下生成正确的目录结构。该测试
无需依赖单仓库 Git 操作即可独立验证。

**Acceptance Scenarios**:

1. **Given** 用户多选 5 个仓库并点击"批量 Clone"，**When** 在配置对话框中选择目标
   根目录、目录组织方式（按平台和账号分组）、HTTPS 协议、并发数 3，**Then** 系统
   应生成 5 个 Clone 任务，任意时刻并发执行不超过 3 个。
2. **Given** Clone 任务正在执行，**When** 后端读取 Git 输出，**Then** 前端应在每个
   任务行实时显示进度百分比、当前阶段（如 "Receiving objects: 65%"）、当前仓库名。
3. **Given** Clone 任务中有 1 个因网络问题失败，**When** 用户点击"重试"，**Then**
   该任务应重新加入队列执行；其他成功任务不受影响。
4. **Given** Clone 任务执行中用户点击"取消"，**When** 取消请求发出，**Then** 该任务
   状态变为 cancelled，对应的子进程被终止，未完成的目标目录被清理。
5. **Given** 目标目录下已存在同名仓库目录，**When** 任务即将执行，**Then** 系统
   应按预设策略跳过该任务并标记 status=skipped，提示用户冲突详情。
6. **Given** 用户选择"按平台和账号分组"的目录组织方式，**When** 克隆完成，**Then**
   目录结构应为 `<root>/<platform>/<owner>/<repo>/`（如 `~/Projects/github/torvalds/linux/`）。
7. **Given** 用户使用 HTTPS 协议克隆私有仓库，**When** 后端需要认证，**Then** 系统
   应通过临时凭据机制注入 Token，**Token 不得写入 remote URL 或持久化日志**；
   克隆完成后 `git remote get-url origin` 的结果应是干净的 URL（不含 token）。
8. **Given** 所有任务执行完成，**When** 用户访问本地仓库页面，**Then** 成功的仓库
   应已自动出现在本地仓库列表中（状态为 clean）。

### User Story 4 - 本地仓库集中管理 (Priority: P2)

作为已经在本地有大量 Git 仓库的开发者，我希望将这些仓库（无论是通过 GitView 克隆
还是手动添加）集中在一个面板中管理，能批量查看每个仓库的工作区状态、批量执行
Fetch 操作，避免频繁切换目录与终端窗口。

**Why this priority**: 本地仓库管理是 US3 克隆产物的承接面，也是 US5 单仓库操作的
入口。先有本地集中视图，再进入单仓库工作区是自然的用户路径。P2 优先级反映其"完
善 MVP 闭环"的定位。

**Independent Test**: 选择本地一个父目录执行"扫描 Git 仓库"，验证扫描出所有 `.git`
目录；对扫描结果执行"刷新状态"，验证每个仓库的当前分支、ahead/behind、变更
文件数被正确读取；多选若干仓库执行"批量 Fetch"，验证全部仓库的远程信息被更新。

**Acceptance Scenarios**:

1. **Given** 用户选择"添加本地仓库"并指定一个有效的 Git 仓库目录，**When** 确认添加，
   **Then** 该仓库应出现在本地仓库列表中，状态字段显示当前分支与未提交变更数量。
2. **Given** 用户选择"扫描目录"并指定一个父目录（如 `~/Projects`），**When** 扫描完成，
   **Then** 该目录下所有包含 `.git` 的子目录应被识别并加入列表（已存在记录不重复添加）。
3. **Given** 本地仓库列表显示 100 个仓库，**When** 用户点击"刷新所有状态"，**Then**
   系统应并行获取所有仓库的状态信息，并在合理时间内完成（不阻塞 UI）。
4. **Given** 仓库工作区有未提交变更，**When** 状态刷新完成，**Then** 列表应显示
   状态标签 "modified" 与变更文件数。
5. **Given** 仓库本地领先远程 2 个 commit、落后 1 个 commit，**When** 状态刷新，
   **Then** 列表应显示 ahead=2、behind=1，状态标签 "need_push"。
6. **Given** 仓库的本地路径已被外部删除，**When** 状态刷新，**Then** 列表应显示
   状态 "path_missing"，并提示用户从列表移除或重新定位。
7. **Given** 用户多选 5 个仓库执行"批量 Fetch"，**When** 任务执行，**Then** 系统应
   显示进度并依次完成；任一仓库失败时不阻塞其他仓库执行。
8. **Given** 用户从本地仓库列表中点击"从列表移除"，**When** 确认操作，**Then** 仅
   从 GitView 的记录中移除（不删除磁盘上的文件）。
9. **Given** 用户对某个仓库点击"打开目录"或"打开终端"，**When** 触发操作，**Then**
   系统应调起操作系统默认的文件管理器或终端，定位到该仓库目录。

### User Story 5 - 单仓库可视化 Git 工作流 (Priority: P2)

作为日常开发者，我希望在 GitView 中完成"查看变更 → stage → commit → push"
完整工作流，无需切换到命令行，从而以更低的认知负担完成日常 Git 操作。

**Why this priority**: 单仓库 Git 工作流是 GitView 作为"Git 客户端"的根本能力。
没有它，GitView 只能算是一个"批量克隆工具"。P2 优先级反映其与 US4 共同构成
MVP 完整闭环的定位。

**Independent Test**: 打开任意已加入本地仓库的仓库，修改若干文件，验证文件变更
列表显示正确状态；选择文件查看 diff，验证差异高亮；stage 一个文件后填入 commit
message 并提交，验证仓库历史增加一条 commit；执行 push 验证远程已更新。

**Acceptance Scenarios**:

1. **Given** 用户从本地仓库列表点击"打开仓库"，**When** 工作区视图加载，**Then** 视图
   应分为三块：顶部仓库信息（仓库名、当前分支、ahead/behind）、左侧文件变更列表、
   右侧 Diff 查看区。
2. **Given** 工作区中有 Modified、Added、Deleted、Untracked 共 4 种状态的文件，
   **When** 文件变更列表渲染，**Then** 每个文件应有对应状态标识（M/A/D/U），并支持
   按状态过滤。
3. **Given** 用户在变更列表中点击某个文件，**When** Diff 加载，**Then** 右侧应显示
   该文件的统一 diff，新增行与删除行以不同颜色高亮。
4. **Given** 用户尝试查看一个二进制文件或大于阈值的大文件，**When** Diff 加载，
   **Then** 系统应显示"二进制文件"或"文件过大不予显示"的提示，而不是卡死 UI。
5. **Given** 用户对若干文件执行 stage，**When** 操作完成，**Then** 这些文件应移动
   到"已暂存"区域；用户可对单个文件或全部已暂存文件执行 unstage。
6. **Given** 用户填写 commit message 与可选的 description，**When** 点击"提交"，
   **Then** 系统应执行提交，工作区状态刷新为 clean，仓库历史增加一条 commit。
7. **Given** 用户未配置 Git user.name 或 user.email，**When** 尝试 commit，**Then**
   系统应阻断提交并提示用户先在设置中配置或全局配置 Git 身份。
8. **Given** 用户多行 commit message 包含特殊字符或换行，**When** 提交，**Then**
   系统应正确处理（不被命令行转义破坏）。
9. **Given** 用户点击 Fetch，**When** 任务执行完成，**Then** 远程分支信息更新，
   ahead/behind 计数同步刷新。
10. **Given** 用户对当前分支点击 Pull，且远程有新提交，**When** 拉取成功，**Then**
    本地分支应快进到远程；如有冲突，应弹出明确的冲突提示（V1 不要求 GUI 解冲突，
    引导用户使用外部工具）。
11. **Given** 用户对已有 upstream 的分支点击 Push，**When** 推送成功，**Then**
    远程应更新；若被拒绝（如 non-fast-forward），应给出明确的中文错误提示。
12. **Given** 用户打开分支列表，**When** 列表渲染，**Then** 应显示本地分支与远程
    分支（区分标识）；用户可点击切换本地分支，或从远程分支 checkout 出新本地分支。

### User Story 6 - 操作日志与问题诊断 (Priority: P3)

作为遇到 Git 操作失败的用户，我希望查看一份清晰的操作日志，了解每次操作的命令、
状态、耗时与错误摘要，并对常见错误获得中文友好的解释，从而能自主定位问题或在
求助时提供准确的上下文。

**Why this priority**: 日志与诊断是质量保障层，对核心功能链路并非阻塞。但对长期
可用性影响重大，故纳入 V1 MVP 而非延后到 V2。P3 优先级反映其"支撑性而非主线"
的定位。

**Independent Test**: 执行若干 Git 操作（Clone、Fetch、Commit、Push），打开操作
日志页面验证每条操作都有记录；故意触发认证失败错误，验证日志中错误信息存在但
Token 已被脱敏；验证常见错误（如 "Authentication failed"）在日志详情中显示中文
友好提示。

**Acceptance Scenarios**:

1. **Given** 用户执行 Clone、Fetch、Pull、Push、Commit、Checkout 等操作，**When**
   操作完成，**Then** 操作日志列表应新增一条记录，包含时间、操作类型、仓库名、
   状态、耗时、命令摘要。
2. **Given** 用户对一个需要认证的 HTTPS 仓库执行 Clone 并使用 Token 认证，**When**
   操作完成，**Then** 日志的命令字段不得出现 Token 字符串；任何形如
   `https://<token>@...` 的 URL 不得出现在日志中。
3. **Given** 用户的 Git 操作因 "Authentication failed" 失败，**When** 用户查看
   日志详情，**Then** 系统除原始错误信息外，应提供中文友好提示
   （如"认证失败，请检查 Token、SSH Key 或账号权限"）。
4. **Given** 用户遇到 "Could not resolve host"、"Permission denied (publickey)"、
   "path already exists and is not an empty directory" 等常见错误，**When** 查看
   日志详情，**Then** 应有对应的中文解释与建议。
5. **Given** 日志记录已积累超过 1 万条，**When** 用户打开日志页面，**Then** 系统应
   分页或虚拟滚动加载，不阻塞 UI。
6. **Given** 用户希望按操作类型筛选日志，**When** 在筛选器中选择 "push"，**Then**
   列表应仅显示 push 类型的日志。

### User Story 7 - 设置与默认目录管理 (Priority: P3)

作为追求可控体验的用户，我希望集中配置 Git 环境路径、默认项目目录、默认 Clone
协议、并发数、主题等偏好，避免每次操作都重复填写。

**Why this priority**: 设置是其他功能的辅助配置面，不直接交付核心价值。但对首次
使用体验至关重要（如 Git 环境检测引导）。P3 优先级反映其支撑性定位。

**Independent Test**: 打开设置页，执行"检测 Git"，验证返回 Git 版本与全局用户
信息；修改默认项目目录、默认 Clone 协议、并发数等设置，关闭并重启应用，验证
设置已持久化生效。

**Acceptance Scenarios**:

1. **Given** 用户首次启动 GitView，**When** 应用执行 Git 环境检测，**Then** 若系统
   未安装 Git，应弹出明确提示并提供安装指引链接；若已安装，应显示 Git 版本。
2. **Given** 用户进入设置页，**When** 查看 Git 设置区，**Then** 应显示当前 Git
   可执行文件路径、全局 user.name、user.email；允许用户修改 Git 可执行文件
   路径并重新检测。
3. **Given** 用户设置默认项目目录为 `~/MyProjects`，**When** 后续使用批量 Clone
   时打开配置对话框，**Then** 目标目录字段应默认填入 `~/MyProjects`。
4. **Given** 用户设置默认 Clone 协议为 SSH、默认并发数为 5、默认目录组织方式为
   "按平台和账号分组"，**When** 后续批量 Clone，**Then** 配置对话框中这些字段
   应自动按设置预填。
5. **Given** 用户在设置中切换主题为"深色"或"跟随系统"，**When** 主题应用，**Then**
   应用界面立即切换；重启后保持上次设置。
6. **Given** 用户配置 HTTP/HTTPS 代理地址，**When** 保存设置，**Then** 后续平台 API
   调用与 Git 操作应通过代理；可选择"使用系统代理"以跟随操作系统设置。
7. **Given** 用户进入"账号与安全"设置区，**When** 查看 Token 凭据状态，**Then**
   应显示每个账号的凭据是否仍存于系统安全存储中，不显示 Token 明文。

### Dashboard 与跨切面验收场景

> 首页仪表盘（FR-059）属于聚合呈现，不构成独立 User Story；但作为首屏核心
> 体验，需独立验证。

1. **Given** 用户已添加 2 个账号、缓存 100 个远程仓库、管理 20 个本地仓库
   （其中 3 个有未提交变更、2 个 ahead、1 个 behind），且有 1 个 Clone 任务
   正在运行，**When** 进入首页仪表盘，**Then** 8 个统计卡片应分别显示：
   账号 2、远程仓库 100、本地仓库 20、运行中 Clone 任务 1、有未提交变更 3、
   需 push 2、需 pull 1、最近失败任务（基于操作日志）。
2. **Given** 用户在仪表盘"最近打开仓库"列表中点击某行的快捷操作（Fetch /
   Pull / Push / 打开目录 / 打开终端），**When** 操作触发，**Then** 应跳转
   到对应执行流程并刷新统计数据。
3. **Given** 用户尚未添加任何账号，**When** 进入首页，**Then** 应显示空状态
   引导卡片，并提供"添加第一个账号"快捷按钮跳转到账号管理页。
4. **Given** 应用因强制关闭中断了若干 Clone 任务，**When** 用户下次启动后进入
   首页，**Then** 这些任务应显示为 interrupted 或 failed 状态（不应仍显示
   "运行中"），与 SC-013 一致。

### Edge Cases

- **Git 未安装**：应用启动时若检测不到 Git，应进入引导模式，提供安装链接；不应
  允许用户进入需要 Git 的功能页面。
- **网络断开/超时**：所有远程操作（API 调用、Clone、Fetch、Pull、Push）必须有明确的
  超时阈值，超时后给出"网络连接失败"的中文提示，并允许重试。
- **API 限流**：当 GitHub / GitLab / Gitee 返回 429 或类似限流响应时，应保留已同步
  数据，提示用户稍后重试，并显示剩余配额（若 API 提供）。
- **Token 过期**：账号 Token 过期时，依赖该账号的所有操作应给出"Token 已过期"提示，
  引导用户重新生成并更新 Token，不应静默失败。
- **目标目录冲突**：批量 Clone 时若目标目录已存在仓库或非空目录，按预设策略
  （跳过 / 提示）处理；不允许"覆盖"模式（V1 安全保护）。
- **大文件 diff**：当文件大小超过阈值（建议 1 MB）或为二进制时，Diff 视图应显示
  提示而非尝试渲染。
- **detached HEAD 状态**：仓库处于 detached HEAD 时，工作区视图应明确显示该状态，
  阻断 commit 操作或要求用户先创建分支。
- **冲突状态**：Pull 后出现冲突时，工作区视图应高亮冲突文件并提示用户使用外部工具
  解决；V1 不要求 GUI 内冲突解决能力。
- **自签名证书**：自建 GitLab 使用自签名证书时，默认不跳过 TLS 校验；用户需在
  实例级高级设置中显式允许，不提供全局跳过开关。
- **内网地址不可达**：内网 GitLab 在公司外网络环境下不可达时，给出"请连接 VPN
  或检查网络"提示，不应反复重试导致 UI 卡死。
- **删除账号的影响**：删除账号时凭据立即从系统安全存储删除；已克隆的本地仓库
  记录保留；远程仓库缓存由用户选择保留或删除。
- **多账号同时同步**：用户同时触发多个账号的"同步仓库"任务时，系统应排队处理，
  避免并发请求导致 API 限流。
- **仓库本地路径外部变更**：仓库在被外部工具（如 `mv`、`rm`）变更后，下次状态
  刷新应识别为 path_missing，引导用户处理而非崩溃。
- **应用关闭时未完成任务**：Clone 等长任务在应用关闭时应标记为 interrupted 或
  failed，避免下次启动后呈现"运行中"的假象。

## Requirements *(mandatory)*

### Functional Requirements

#### 账号管理（对应 US1）

- **FR-001**: 系统 MUST 支持添加 GitHub、GitLab、Gitee 三大平台账号，账号信息包括
  平台类型、Web 地址、API 地址、用户名、显示名、头像、Token 凭据键。
- **FR-002**: 系统 MUST 支持同一平台下多个账号并列存放，互不覆盖，由用户标记其中
  一个为默认账号。
- **FR-003**: 系统 MUST 在用户填写 Token 后提供"测试连接"能力，验证 Token 有效性、
  API 地址正确性、账号权限充足性，并返回结构化的错误分类。
- **FR-004**: 系统 MUST 支持自建 GitLab 实例配置，至少覆盖：自定义 Web/API 地址、
  非标准端口、HTTP/HTTPS 协议、内网域名、自签名证书白名单、HTTPS/SSH 默认协议、
  代理设置、SSH 主机别名。
- **FR-005**: 系统 MUST 在用户填写 GitLab Web 地址后自动推导 API 地址（追加 `/api/v4`），
  并允许手动修改。
- **FR-006**: 系统 MUST 在使用 HTTP 明文连接或自签名证书时显示安全提示，需用户显式
  确认才能保存。
- **FR-007**: 系统 MUST 将 Token 存储在操作系统原生安全凭据服务中，禁止以明文形式
  存储于本地数据库或配置文件。
- **FR-008**: 系统 MUST 在删除账号时同步删除安全凭据服务中对应的 Token，保留已
  克隆的本地仓库记录。
- **FR-009**: 系统 MUST 允许用户编辑账号备注、启用/禁用账号、设为默认账号。

#### 远程仓库浏览（对应 US2）

- **FR-010**: 系统 MUST 支持按账号、按平台或全量同步远程仓库列表，同步过程中
  显示进度（已同步 N/M）。
- **FR-011**: 系统 MUST 在同步过程中处理 API 分页与限流，保证账号下所有可见仓库
  完整入库；遇限流时保留已同步数据并提示重试。
- **FR-012**: 系统 MUST 在同步失败时保留旧缓存，不清空已有仓库数据。
- **FR-013**: 系统 MUST 缓存远程仓库元数据（名称、Owner、描述、默认分支、可见性、
  Web URL、HTTPS/SSH Clone URL、语言、最近更新时间）到本地存储，列表默认从缓存
  渲染。
- **FR-014**: 系统 MUST 提供仓库搜索能力，至少支持仓库名与描述的模糊匹配。
- **FR-015**: 系统 MUST 提供筛选能力，包括平台、账号、Owner/Group、可见性
  (public/private/internal)、是否已克隆、是否已收藏；多筛选条件可组合。
- **FR-016**: 系统 MUST 支持仓库收藏与取消收藏，收藏状态持久化保存。
- **FR-017**: 系统 MUST 提供仓库详情视图，展示完整元信息与本地克隆状态；提供
  打开网页、复制 HTTPS/SSH URL、打开本地仓库（若已克隆）等快捷操作。

#### 批量克隆（对应 US3）

- **FR-018**: 系统 MUST 支持在远程仓库列表中多选若干仓库，并通过"批量 Clone"
  入口下发任务。
- **FR-019**: 系统 MUST 在批量 Clone 配置对话框中允许用户指定：目标根目录、目录
  组织方式（扁平/按平台和账号分组/按 Owner 分组）、Clone 协议（HTTPS/SSH）、并发
  数量、已存在目录处理策略、克隆完成后是否自动加入本地仓库列表。
- **FR-020**: 系统 MUST 按用户配置的并发数限制同时执行的 Clone 任务数量（默认 3，
  上限不超过预设安全阈值如 8）。
- **FR-021**: 系统 MUST 实时解析每个 Clone 任务的进度，包括百分比、当前阶段
  (Enumerating/Counting/Compressing/Receiving/Resolving/Updating 等)。
- **FR-022**: 系统 MUST 支持以下任务状态：pending、running、success、failed、
  cancelled、skipped；每种状态对应明确的 UI 标识与可执行操作。
- **FR-023**: 系统 MUST 支持单任务取消（终止子进程并清理半成品目录）、单任务重试、
  打开任务对应的目标目录、查看任务日志、清空已完成任务。
- **FR-024**: 系统 MUST 在使用 HTTPS 协议克隆私有仓库时通过临时凭据机制注入认证
  信息（如临时 askpass、临时 credential helper 或环境变量），**禁止**将 Token
  写入 remote URL 或持久化日志；克隆完成后本地 `remote.origin.url` 必须为
  干净 URL。
- **FR-025**: 系统 MUST 在批量 Clone 成功后自动将仓库加入本地仓库列表（前提是
  用户启用"克隆完成后自动添加"选项）。

#### 本地仓库管理（对应 US4）

- **FR-026**: 系统 MUST 支持三种方式将本地仓库纳入管理：手动选择单个 Git 仓库
  目录、选择父目录扫描其下所有 Git 仓库、Clone 成功后自动加入。
- **FR-027**: 系统 MUST 在扫描时识别 `.git` 目录的存在并跳过已加入列表的仓库，
  避免重复记录。
- **FR-028**: 系统 MUST 维护每个本地仓库的状态：当前分支、默认分支、远程 URL、
  所属账号、ahead/behind 计数、未提交文件数、最后打开时间、最后状态检查时间。
- **FR-029**: 系统 MUST 支持以下仓库状态枚举:clean、modified、need_push、
  need_pull、conflict、detached_head、no_remote、path_missing；每种状态有对应
  UI 标识。
- **FR-030**: 系统 MUST 提供单仓库刷新状态与"刷新所有仓库状态"能力，刷新操作
  必须异步执行，不阻塞 UI。
- **FR-031**: 系统 MUST 提供多仓库批量 Fetch 能力，任一仓库失败不阻塞其他仓库
  执行，所有结果汇总展示。
- **FR-032**: 系统 MUST 提供"从列表移除"操作，仅删除 GitView 内部记录，不删除
  磁盘文件；操作执行前需要用户显式确认。
- **FR-033**: 系统 MUST 提供"打开目录"与"打开终端"快捷操作，调起操作系统默认
  的文件管理器或终端，定位到该仓库目录。

#### 单仓库工作区（对应 US5）

- **FR-034**: 系统 MUST 提供仓库工作区视图，包含：顶部仓库信息栏（仓库名、当前
  分支、ahead/behind、远程 URL、快捷操作按钮）、文件变更列表、Diff 查看区、
  提交面板。
- **FR-035**: 系统 MUST 在文件变更列表中区分 Modified、Added、Deleted、Renamed、
  Untracked、Conflict 六种文件状态，并支持按状态过滤。
- **FR-036**: 系统 MUST 支持以下文件级操作：stage 单文件、unstage 单文件、stage all、
  unstage all、discard changes（需二次确认）、打开文件、在文件管理器中显示。
- **FR-037**: 系统 MUST 提供 Diff 查看能力，显示统一 diff 格式（unified diff），
  新增行与删除行以不同颜色高亮；对二进制文件、超大文件、图片文件给出占位提示
  而非尝试渲染。
- **FR-038**: 系统 MUST 提供 commit 面板，包含 commit message 与可选的 description
  字段；提交前校验已暂存文件存在、message 非空、Git user.name/user.email 已配置、
  非冲突状态、非 detached HEAD。
- **FR-039**: 系统 MUST 支持多行 commit message 与特殊字符提交，通过临时文件
  方式避免命令行转义问题。
- **FR-040**: 系统 MUST 提供顶部 Fetch、Pull、Push 快捷按钮，操作完成后刷新仓库
  状态。
- **FR-041**: 系统 MUST 在 Pull 出现冲突时明确提示用户，并标识冲突文件；V1 不
  要求 GUI 内解决冲突。
- **FR-042**: 系统 MUST 在 Push 被拒绝（如 non-fast-forward）时给出中文错误提示
  与建议（如"请先 Pull 远程更新"）。
- **FR-043**: 系统 MUST 提供分支列表能力，区分显示本地分支与远程分支；支持切换
  本地分支、从远程分支 checkout 新本地分支。
- **FR-044**: 系统 MUST 阻断在脏工作区（有未提交变更）下的分支切换并给出提示，
  除非用户先 stash 或提交。

#### 操作日志与诊断（对应 US6）

- **FR-045**: 系统 MUST 记录所有 Git 操作（clone、fetch、pull、push、commit、
  checkout、branch、merge、rebase）与平台 API 请求错误、账号验证、仓库扫描的
  操作日志。
- **FR-046**: 日志记录字段 MUST 包括：时间、操作类型、仓库名（若适用）、命令
  摘要、状态、耗时、输出摘要、错误摘要。
- **FR-047**: 系统 MUST 对日志中的敏感信息做脱敏，包括：Token、密码、OAuth
  refresh token、SSH 私钥内容、带 token 的 Clone URL；脱敏后日志可安全展示
  与分享。
- **FR-048**: 系统 MUST 对常见 Git 错误提供中文友好提示，至少覆盖：Authentication
  failed、Repository not found、Permission denied (publickey)、Could not resolve
  host、path already exists and is not an empty directory。
- **FR-049**: 系统 MUST 提供日志列表的分页或虚拟滚动能力，支持 10000+ 条日志
  不阻塞 UI。
- **FR-050**: 系统 MUST 支持按操作类型、状态、仓库筛选日志。

#### 设置中心（对应 US7）

- **FR-051**: 系统 MUST 在首次启动时检测 Git 可执行文件存在性与版本号；未安装
  Git 时阻断进入 Git 相关功能页面，提供安装引导。
- **FR-052**: 系统 MUST 提供 Git 设置界面，允许查看与修改：Git 可执行文件路径、
  Git user.name、Git user.email、默认 pull 策略、默认 push 策略。
- **FR-053**: 系统 MUST 提供通用设置，包括：启动时打开上次仓库、启动时自动检查
  仓库状态、默认项目目录、默认 Clone 协议、默认 Clone 并发数、默认目录组织方式、
  语言（中文/英文）、主题（浅色/深色/跟随系统）。
- **FR-054**: 系统 MUST 提供网络设置，包括：HTTP/HTTPS 代理、使用系统代理开关、
  API 请求超时、Clone 超时。
- **FR-055**: 系统 MUST 提供"账号与安全"设置面板，显示每个账号的凭据存储状态，
  允许重新验证、删除凭据；不显示 Token 明文。
- **FR-056**: 系统 MUST 持久化所有用户设置，应用重启后设置保持生效。

#### 跨切面要求

- **FR-057**: 系统 MUST 在长耗时任务（远程同步、批量 Clone、单仓库 Fetch/Pull/Push、
  仓库状态批量刷新）执行时不阻塞 UI 主线程，前端始终保持响应。
- **FR-058**: 系统 MUST 在所有远程操作（API、Git 网络命令）中支持代理配置生效。
- **FR-059**: 系统 MUST 提供首页仪表盘视图，聚合展示账号数、远程仓库数、本地仓库数、
  运行中任务数、有未提交变更的仓库数、最近打开仓库、最近任务等概览信息。

### Key Entities *(include if feature involves data)*

- **账号 (Account)**：代表一个平台用户身份。关键属性：平台类型（GitHub/GitLab/Gitee）、
  Web 地址、API 地址、用户名、显示名、头像 URL、凭据键、是否默认、是否启用、
  创建时间、最近同步时间。一个账号关联多个远程仓库。
- **GitLab 实例配置 (GitLab Instance Config)**：自建 GitLab 账号的扩展配置。关键
  属性：实例名称、Web/API 地址、是否允许 HTTP 明文、是否允许自签名证书、是否使用
  系统代理、自定义代理 URL、请求超时、默认 Clone 协议、SSH 主机别名、API 路径前缀。
  一对一关联到账号。
- **远程仓库 (Remote Repository)**：账号下可见的远程仓库元数据缓存。关键属性：
  所属账号、平台、Owner/Group、仓库名、完整名、描述、默认分支、可见性、Web URL、
  HTTPS Clone URL、SSH Clone URL、语言、最近更新时间、最近同步时间、是否收藏、
  本地仓库关联 ID（若已克隆）。
- **本地仓库 (Local Repository)**：纳入 GitView 管理的本地 Git 仓库。关键属性：
  仓库名、本地路径、远程 URL、所属平台/账号、Owner、当前分支、默认分支、状态
  枚举、ahead/behind 计数、变更文件数、最后打开时间、最后状态检查时间。
- **克隆任务 (Clone Task)**：批量 Clone 中的单个任务。关键属性：关联的远程仓库
  与账号、仓库名、远程 URL、目标路径、状态、进度百分比、当前阶段、错误信息、
  创建/开始/结束时间。
- **操作日志 (Operation Log)**：用户或系统执行的操作记录。关键属性：操作类型、
  关联仓库 ID（若适用）、命令摘要（已脱敏）、状态、输出摘要、错误摘要、耗时、
  创建时间。
- **设置 (Settings)**：用户偏好的键值对集合。键覆盖通用设置、Git 设置、网络
  设置、外部工具设置等。

## Success Criteria *(mandatory)*

### Measurable Outcomes

#### 用户效率与体验

- **SC-001**: 新用户能在 **5 分钟内**完成从"添加第一个账号"到"成功克隆一个仓库"
  的全流程（不计 Git 实际下载时间）。
- **SC-002**: 已有大量本地仓库的用户能在 **30 秒内**通过"扫描目录"将 50 个本地
  仓库一次性纳入管理。
- **SC-003**: 用户能在 **3 次点击内**完成多选 10 个远程仓库并启动批量 Clone 的
  全部操作。
- **SC-004**: 用户在远程仓库列表中输入搜索关键词后，过滤结果应在 **500 毫秒内**
  呈现（前提：列表数据已缓存）。

#### 规模与性能

- **SC-005**: 本地仓库列表 MUST 稳定支持 **至少 500 个仓库**的展示与状态刷新，
  UI 不卡顿、不崩溃。
- **SC-006**: 远程仓库列表 MUST 稳定支持 **至少 5000 条仓库数据**的分页或虚拟
  滚动展示。
- **SC-007**: 操作日志 MUST 稳定支持 **至少 10000 条记录**的浏览与筛选。
- **SC-008**: 批量 Clone 在默认并发数 3、网络带宽充足的前提下，**10 个中型仓库
  （单仓库 ≤ 50 MB）应在 5 分钟内全部完成**（不计网络异常时间）。

#### 安全与合规

- **SC-009**: Token 在本地数据库与所有日志中的明文出现次数 = **0**（可通过自动
  化扫描验证）。
- **SC-010**: 任何带 token 的 URL（形如 `https://<token>@...`）在日志、UI、本地
  `remote.origin.url` 中出现次数 = **0**。
- **SC-011**: 用户删除账号后，**100% 的关联凭据**应从操作系统安全存储中删除
  （可通过凭据查询命令验证）。

#### 稳定性与可恢复性

- **SC-012**: 单个 Git 操作失败 **不得**导致应用崩溃；MVP 验收阶段崩溃率应低于
  每千次操作 1 次。
- **SC-013**: 应用强制关闭后重启，**100% 未完成的 Clone 任务**应被标记为
  interrupted 或 failed，不存在"运行中"幻象。
- **SC-014**: 远程仓库同步失败时 **0** 已缓存数据被清空。

#### 跨平台一致性

- **SC-015**: 全部 MVP 验收用例（详见下文）在 **macOS、Windows、Ubuntu 三个平台
  上均通过**。
- **SC-016**: 跨平台界面布局与文本呈现一致性 ≥ **95%**（仅允许平台原生控件差异）。

#### 错误诊断有效性

- **SC-017**: 至少 **5 类常见错误**（认证失败、仓库不存在、SSH 公钥拒绝、DNS 解析
  失败、目标目录冲突）在用户视角下提供中文友好提示，且提示中包含明确的下一步
  建议。

#### MVP 闭环验证

- **SC-018**: 以下 MVP 验收用例在 V1 发布前必须 100% 通过：
  1. 用户可以成功启动应用。
  2. 应用可以检测系统 Git 是否安装。
  3. 用户可以添加 GitHub 账号并同步仓库。
  4. 用户可以添加 GitLab（含自建实例）账号并同步仓库。
  5. 用户可以添加 Gitee 账号并同步仓库。
  6. 用户可以在远程仓库中心搜索和筛选仓库。
  7. 用户可以多选多个仓库并批量 Clone。
  8. Clone 任务可以显示进度。
  9. Clone 失败后可以重试。
  10. Clone 成功后仓库进入本地仓库列表。
  11. 用户可以手动添加本地 Git 仓库。
  12. 用户可以查看本地仓库状态。
  13. 用户可以查看文件变更和 diff。
  14. 用户可以 stage / unstage 文件。
  15. 用户可以提交 commit。
  16. 用户可以执行 fetch / pull / push。
  17. 用户可以查看分支并切换分支。
  18. 用户可以查看操作日志。
  19. Token 不会出现在数据库明文字段中。
  20. Token 不会出现在日志中。

## Assumptions

- **用户具备基础 Git 知识**：用户理解 commit、branch、push、pull 等概念；GitView
  不承担 Git 教学职责。
- **用户已安装 Git CLI**：V1 不内置 Git 二进制；GitView 依赖系统 Git CLI 完成
  实际 Git 操作。未安装时给出引导。
- **用户能自行管理 Token 与 SSH Key**：V1 使用 Personal Access Token 登录平台，
  Token 由用户在各平台自行生成；SSH Key 配置由用户在 `~/.ssh/config` 中自行
  维护，GitView 不提供 SSH Key 管理 UI。
- **平台 API 稳定**：假设 GitHub、GitLab、Gitee 的 REST API 在 V1 开发期内行为
  与文档一致；遇到限流时以平台规则为准。
- **桌面环境支持原生凭据存储**：假设 macOS、Windows、Linux 桌面环境提供可用的
  原生安全凭据存储；Linux 环境若 Secret Service 不可用，提示用户安装相关依赖。
- **网络环境合理**：假设用户网络可访问目标平台（GitHub 国内访问异常等环境问题
  通过代理设置解决，不在 V1 范围内提供加速能力）。
- **单用户单设备**：V1 不考虑多用户协作、多设备同步设置等场景；所有数据存储在
  本地。
- **V1 仅支持 PAT 认证**：OAuth 登录、SSO 登录、双因素动态码等不在 V1 范围；
  V2/V3 可扩展。
- **V1 不内置冲突解决**：Git 合并冲突需用户使用外部工具解决；V1 只检测与提示。
- **V1 不实现 PR/MR、Issue、CI 等平台协作能力**：仅提供"打开网页"快捷入口跳转到
  对应平台页面；完整协作能力归入 V3。
- **V1 不实现 stash、tag、merge、rebase、cherry-pick、reset、commit graph、
  AI 辅助等进阶能力**：归入 V2 或 V4。
- **目录组织方式默认"按平台和账号分组"**：除非用户在批量 Clone 配置中明确选择
  其他方式。
- **默认并发数 3**：除非用户显式调整；最大上限不超过 8（性能与稳定性平衡）。
