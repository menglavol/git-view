# GitView 性能基线报告

记录 spec.md 的规模与性能指标(SC-005 ~ SC-008)的实测结果。本文件为**模板**，实测数据需在目标环境按下方步骤采集后填入。

## 复现步骤

```bash
# 1. 正常启动一次应用以创建数据库（或用 GITVIEW_DB 指定已存在的库）
# 2. 写入基线种子数据（本地 500 / 远程 5000 / 日志 10000）
bash scripts/perf/seed-data.sh

# 3. 测量数据层查询耗时（SC-006 / SC-007 的 SQL 部分）
node scripts/perf/measure.mjs

# 4. UI 渲染、状态批量刷新、批量 Clone 等需在应用内手动计时（见下表）
# 5. 测完清理种子数据
bash scripts/perf/seed-data.sh --clean
```

## 指标与结果

| 指标 | 目标 | 数据规模 | 测量方式 | 实测 | 结论 |
| --- | --- | --- | --- | --- | --- |
| **SC-005** 本地仓库列表展示与状态刷新 | UI 不卡顿、不崩溃 | 500 本地仓库 | 应用内：打开本地仓库页 + 「刷新所有状态」计时 | _待填_ | _待填_ |
| **SC-006** 远程仓库列表与搜索 | 虚拟滚动流畅；搜索 ≤ 500ms(已缓存) | 5000 远程仓库 | `measure.mjs` 查询耗时 + 应用内搜索响应计时 | _待填_ | _待填_ |
| **SC-007** 操作日志浏览与筛选 | 10000 条不阻塞 | 10000 日志 | `measure.mjs` 分页/筛选耗时 + 应用内滚动 | _待填_ | _待填_ |
| **SC-008** 批量 Clone | 10 个中型仓库(≤50MB)并发 3 在 5 分钟内完成 | 10 仓库 | 应用内：批量 Clone 总耗时(不计网络异常) | _待填_ | _待填_ |

> 补充参考目标(plan.md)：冷启动 ≤ 3s；搜索过滤 ≤ 500ms；主线程无响应 ≤ 200ms；空闲内存 < 200MB。

## 偏差与说明

- `scripts/perf/measure.mjs` 仅覆盖**数据层 SQL** 耗时；UI 渲染流畅度依赖前端虚拟滚动，需肉眼+录屏评估。
- 批量 Clone(SC-008)耗时受网络带宽影响大，应在带宽充足、排除网络异常后取值。
- 若某项不达标，请在「结论」列记录实测值与差距，并在此处补充分析与优化措施。
